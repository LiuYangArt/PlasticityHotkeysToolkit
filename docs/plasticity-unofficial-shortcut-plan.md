# Plasticity 非官方快捷键扩展实施计划

## 目标

建立一套**不依赖官方公开命令**的快捷键扩展方案，用来给 Plasticity 补充官方 `keymap.json` 无法覆盖的动作。

本计划以以下目标动作作为样板：

- 在左侧文件夹面板中，选中文件夹后按 `Enter`，把该文件夹设为 `Active Folder`

## 预期产物

最终方案建议包含以下组成：

1. 一份我们自己的 overlay 配置文件
2. 一套运行时注入机制
3. 一层自定义命令适配器
4. 一组针对具体 UI 动作的桥接实现

建议产物命名：

- `custom-shortcuts.json`
- `injector.js`
- `shortcut-adapters.js`
- 可选的 `launcher.ps1` 或其他启动器

## 成功标准

### 第一阶段成功标准

- 能在不改 Plasticity 安装包的前提下，把自定义脚本稳定注入到渲染层。
- 能在左侧文件夹面板聚焦时，监听到 `Enter`。
- 能识别当前选中的文件夹条目。

### 第二阶段成功标准

- `Enter` 可以复用当前“双击设为 Active Folder”的现有行为。
- 不影响官方已存在的快捷键。
- 如果焦点不在目标面板，`Enter` 不应误触发。

### 第三阶段成功标准

- 可以通过我们自己的 overlay 配置文件继续添加更多非官方快捷键。
- 快捷键规则具备最小可维护性，不需要每次都手改脚本逻辑。

## 非目标

当前计划不包含以下内容：

- 不做 Plasticity 内部命令系统的完整逆向。
- 不做 `.jsc` 深度 patch 作为第一版方案。
- 不追求把所有非官方动作都映射成官方 `keymap.json` 可识别的命令。
- 不把 OS 层宏自动化当作最终长期方案。

## 推荐架构

推荐使用“四层结构”：

```text
custom-shortcuts.json
  -> injector runtime
  -> custom command adapters
  -> Plasticity existing UI behavior
```

更具体的职责划分如下：

### 1. Overlay 配置层

用途：

- 维护我们自己的快捷键定义。
- 与官方 `keymap.json` 分离，避免混淆。

建议形式：

```json
{
  "body:not([gizmo]) .plasticity-assets": {
    "enter": "custom:assets:set-active-folder"
  }
}
```

### 2. 注入层

用途：

- 把我们的脚本放进 Plasticity 渲染层。
- 负责监听键盘事件、匹配上下文、调用适配器。

实现职责：

- 监听 `keydown`
- 归一化键位
- 判断是否在目标 selector 范围内
- 转发到对应自定义命令

### 3. 适配器层

用途：

- 处理“官方未公开命令”的动作桥接。
- 把抽象命令翻译为具体 UI 行为。

例如：

- `custom:assets:set-active-folder`
- `custom:outliner:toggle-expanded`
- `custom:assets:open-selected`

### 4. Plasticity 行为复用层

用途：

- 尽量复用 Plasticity 已存在的交互，不重写业务语义。

优先顺序：

1. 复用 DOM 双击行为
2. 复用内部组件回调
3. 最后才考虑 patch 内部逻辑

## 阶段拆解

## Phase 0: 启动与附着策略验证

目标：

- 先证明“我们能把脚本稳定注入到 Plasticity 渲染层”。

任务：

1. 验证 Plasticity 是否支持通过运行参数开启可附着调试端口。
2. 验证能否在启动后连接到渲染进程并执行 `Runtime.evaluate` 类似注入。
3. 如果运行时注入不可行，再评估启动器包裹或其他非侵入附着方案。

完成标准：

- 可以让一段最小脚本在渲染层打印标记、监听按键或改动一个无害标记。

风险门槛：

- 如果连稳定注入都做不到，就不要继续做自定义命令层，先重新选附着方案。

## Phase 1: UI 上下文探测

目标：

- 搞清楚目标面板的 DOM、选中态和焦点行为。

任务：

1. 确认左侧文件夹区域对应的根节点和可匹配 selector。
2. 确认当前选中项的 DOM 特征。
3. 确认 folder 项与其他 asset 项是否可区分。
4. 确认按键监听应挂在 document、面板容器还是更局部节点。

完成标准：

- 能稳定回答三个问题：
  - 当前是否位于文件夹面板上下文
  - 当前是否存在选中项
  - 当前选中项是否为 folder

## Phase 2: 激活动作桥接

目标：

- 让我们的自定义命令能够真正触发“Set Active Folder”。

优先实现路径：

### 方案 A：派发合成双击

步骤：

1. 找到当前选中条目的真实 DOM。
2. 构造与鼠标双击等价的事件序列。
3. 验证是否触发原生“设为 Active Folder”行为。

验收标准：

- 与人工双击结果一致。

退出条件：

- 如果 UI 明显更新且行为一致，就保留该方案。

### 方案 B：调用内部激活链

触发条件：

- 如果方案 A 失败，或双击不是标准 DOM 事件驱动。

步骤：

1. 继续围绕 `handleActivate` / `onActivate` 做内部链路定位。
2. 找到当前选中项对象与激活入口。
3. 由适配器直接调用该逻辑。

验收标准：

- 不依赖鼠标事件模拟，也能得到完全一致的结果。

## Phase 3: 建立自定义快捷键 overlay

目标：

- 让“补快捷键”从单点 hack 变成可维护机制。

任务：

1. 设计 `custom-shortcuts.json` 的最小格式。
2. 实现 chord 归一化逻辑。
3. 实现 selector/context 匹配逻辑。
4. 实现自定义命令注册表。

建议数据结构：

```json
{
  "body:not([gizmo]) .plasticity-assets": {
    "enter": "custom:assets:set-active-folder",
    "shift-enter": "custom:assets:open-selected"
  }
}
```

验收标准：

- 后续新增一个非官方快捷键，不需要改注入主流程，只需要补配置和适配器。

## Phase 4: 冲突处理与安全护栏

目标：

- 避免非官方快捷键破坏已有交互。

任务：

1. 只在目标上下文拦截按键。
2. 只在选中项满足条件时触发。
3. 对输入框、重命名状态、搜索框状态直接放行。
4. 如果官方未来新增同名快捷键，要有覆盖优先级规则。

建议规则：

- 输入态优先于快捷键态。
- 官方已有命令优先于自定义桥接。
- 自定义桥接必须显式 `preventDefault`，不能模糊吞键。

## Phase 5: 持久化与启动方式

目标：

- 让这套能力可以长期使用，而不是每次手工注入。

候选方案：

### 方案 1：自定义启动器

做法：

- 用我们自己的启动脚本启动 Plasticity。
- 启动后自动连接并注入脚本。

优点：

- 不改安装目录。
- 升级后容易恢复。

缺点：

- 用户必须通过启动器进入程序。

### 方案 2：辅助常驻进程

做法：

- 单独运行一个 helper。
- 监听 Plasticity 进程出现后完成注入。

优点：

- 使用体验更接近“自动生效”。

缺点：

- 生命周期管理更复杂。

### 方案 3：安装包 patch

做法：

- 直接改 Plasticity 的加载逻辑或字节码。

结论：

- 作为最后手段保留，不作为首期实施路线。

## 关键风险

### 风险 1：注入入口不稳定

影响：

- 可能根本没法把我们的 JS 放进渲染层。

应对：

- 把 Phase 0 作为第一道关卡，先做最小验证。

### 风险 2：双击逻辑不是 DOM 驱动

影响：

- 合成 `dblclick` 不一定能复用原有行为。

应对：

- 预先准备方案 B，继续逆向 `handleActivate / onActivate` 链。

### 风险 3：选中态难以稳定识别

影响：

- 可能分不清当前选中的是 folder 还是其他项目。

应对：

- 在 Phase 1 优先确定 DOM 结构与状态特征，不要过早写快捷键逻辑。

### 风险 4：升级后失效

影响：

- UI 类名、DOM 结构、内部状态名都可能变化。

应对：

- 把适配器做成小而可替换的模块。
- 不把路径绑定到单个脆弱节点上。

## 验证清单

进入实现阶段后，建议按下面顺序验收：

1. 注入脚本能稳定执行。
2. `Enter` 只在目标面板焦点下被捕获。
3. 输入框和搜索状态下不会误触发。
4. 当前选中 folder 时，`Enter` 与双击结果一致。
5. 当前选中非 folder 时，不会误触发。
6. 官方已有快捷键不受影响。
7. 应用重启后可重复生效。

## 首个样板功能的建议落地顺序

建议先只做这一个动作：

```text
Enter -> custom:assets:set-active-folder
```

最小落地顺序如下：

1. 打通注入。
2. 识别当前选中 folder。
3. 尝试合成双击。
4. 如果失败，转内部回调桥接。
5. 将该动作抽象成一个 `custom:` 命令。
6. 再把配置读取抽出来，形成通用 overlay。

这样做的原因是：

- 先证明“桥接一个未公开动作”是可行的。
- 再考虑把这条路径抽象成通用快捷键扩展框架。

## 最终建议

不要把第一版目标设成“彻底接管 Plasticity 快捷键系统”。  
更务实的路线是：

1. 先做一个样板动作。
2. 证明渲染层桥接可行。
3. 再把桥接能力抽象成我们自己的非官方快捷键系统。

如果这个样板能跑通，后续再给更多“官方未暴露命令”的动作补快捷键，成本会显著下降。
